# 前言
**[[数据密集型]]**： 数据是其主要挑战（数据量，数据复杂度或数据变化速度）。
**[[计算密集型]]**：处理器速度是其瓶颈。
本书的目的：尝试寻找 **思考** 数据系统的有效方式 —— 不仅关于它们 **如何** 工作，还包括它们 **为什么** 以这种方式工作，以及哪些问题是我们需要问的。阅读本书后，你能很好地决定哪种技术适合哪种用途，并了解如何将工具组合起来，为一个良好应用架构奠定基础。

# 第一部分：数据系统基础
# 1.数据系统架构中的**权衡**
- [[数据密集型]]：
	- 存储数据，以便它们或其他应用程序以后能再次找到（**数据库**）
	- 记住昂贵操作的结果，以加快读取速度（**缓存**）
	- 允许用户按关键字搜索数据或以各种方式过滤数据（**搜索索引**）
	- 一旦事件和数据变更发生就立即处理（**流处理**）
	- 定期处理累积的大量数据（**批处理**）
- [[后端]]服务通常通过 [[HTTP]]（有时是 WebSocket）提供访问。其核心是应用代码：在一个或多个数据库中读写数据，并按需接入缓存、消息队列等其他系统（可统称为 **数据基础设施**）。应用代码往往是 **无状态** 的：处理完一个 HTTP 请求后，不保留该请求上下文。因此，凡是需要跨请求持久化的信息，都必须写在客户端，或写入服务器端数据基础设施。
### 分析型与事务型系统
- **[[事务型系统]]** 由后端服务和数据基础设施组成，在这里创建数据，例如通过服务外部用户。在这里，应用程序代码基于用户执行的操作读取和修改其数据库中的数据。
- **[[分析型系统]]** 服务于业务分析师和数据科学家的需求。它们包含来自事务型系统的只读数据副本，并针对分析所需的数据处理类型进行了优化。
### 事务处理与分析的特征
- **[[联机事务处理]]**：基于用户的输入插入、更新或删除记录。
- **[[联机分析处理]]**：扫描大量记录，并计算聚合统计信息（如计数、求和或平均值），而不是将单个记录返回给用户。
- 表 1-1. 事务型系统和分析型系统特征比较

|属性|事务型系统（OLTP）|分析型系统（OLAP）|
|---|---|---|
|主要读取模式|点查询（通过键获取单个记录）|对大量记录进行聚合|
|主要写入模式|创建、更新和删除单个记录|批量导入（ETL）或事件流|
|人类用户示例|Web 或移动应用程序的最终用户|内部分析师，用于决策支持|
|机器使用示例|检查操作是否被授权|检测欺诈/滥用模式|
|查询类型|固定的查询集，由应用程序预定义|分析师可以进行任意查询|
|数据代表|数据的最新状态（当前时间点）|随时间发生的事件历史|
|数据集大小|GB 到 TB|TB 到 PB|
### 数据仓库
**一个专门用于分析的独立数据库**，它通过 **ETL（提取-转换-加载）** 过程，从企业分散的各类事务处理系统（OLTP）中获取数据的只读副本，并进行清洗和转换，使分析师可以在不影响公司日常业务操作的情况下，方便地查询和分析来自不同源系统的数据，以支持决策。
![](https://ddia.vonng.com/fig/ddia_0101.png)

图 1-1. ETL 到数据仓库的简化概述。
**OLTP 采用的“实体-关系模型”（3NF，即第三范式）是为了高效处理事务**，它通过将数据拆解成大量离散的实体表（如用户、订单、产品、地址），并用外键关联，来确保数据无冗余、更新一致、插入删除快。但这恰恰是分析的“死穴”——分析查询往往需要**聚合、分组、排序**，意味着要同时关联（JOIN）很多张表，扫描海量数据行。表越多、数据量越大，这类查询就越慢、越复杂，甚至无法在合理时间内完成。
而数据仓库采用的“[[星型模型]]”和“[[雪花型模型]]”就是为了分析而设计的：
- **星型模型**：将数据组织成一个 **“事实表”**（记录业务事件，如每一笔销售）和多个 **“维度表”**（记录业务对象，如时间、产品、门店）。事实表通过外键链接维度表。这种模式是非规范化的，维度表里会包含很多冗余的描述信息（如产品名称、类别、品牌都放在一张产品表里）。
    - **为什么适合分析？** 因为一个典型的分析查询（如“按产品类别统计去年每个季度的总销售额”）只需要关联**一张事实表**和**一到两张维度表**。JOIN 数量少，查询更简单、更快。
- **雪花型模型**：是星型模型的变种，它会进一步将维度表**规范化**，拆分成更细的子维度表。例如，将产品维度表拆分成产品、品牌、类别三张独立的表。
    - **为什么存在？** 雪花型比星型**节省存储空间**，并且对某些维度层次变化的维护更友好。
    - **为什么不是主流？** 对于分析查询来说，雪花型引入了**更多的关联操作**，查询性能通常不如星型模型。因此，在追求查询速度和简洁性的现代数据仓库（如 Google BigQuery, Amazon Redshift）中，星型模型（甚至更宽、更非规范化的表）更为常见。
#### 从数据仓库到数据湖
- **数据湖**：一个集中式存储库，用于存放任何类型（不限于关系型数据）的原始数据副本。
	- **无强制模式**：与数据仓库不同，数据湖不强制特定的文件格式或数据模型，可以存储文本、图像、视频等任意格式的数据。
	- **成本较低**：通常基于廉价的对象存储（如云存储）构建。
	- **面向数据科学**：主要为了满足数据科学家对**特征工程**、**自然语言处理**等非SQL任务的需求。
	- **遵循“寿司原则”**：保存最原始的数据，让每个数据消费者根据自己的需要进行转换。
- **数据湖仓**：结合了数据湖的灵活性和数据仓库的分析能力，允许直接在数据湖文件上运行SQL查询。
- **传统 ETL** x **数据管道**
	-  **传统 ETL**：是数据管道的一种**特定、早期的实现形式**，主要服务于数据仓库。
	- **数据管道**：是一个**更通用、更现代的概念**，覆盖了从简单数据复制到复杂实时流处理的各种数据移动和处理流程，是构建数据湖和数据湖仓的核心组件。

### 记录系统与派生数据
- **权威记录系统**：记录系统，也称 **真相来源（权威数据源）**，保存某类数据的权威（canonical）版本。新数据进入系统时（例如用户输入）首先写入这里。每个事实只表示一次（这种表示通常是 **规范化** 的；见[“规范化、反规范化与连接”](https://ddia.vonng.com/ch3/#sec_datamodels_normalization)）。如果其他系统与记录系统不一致，则按定义以记录系统为准。
- **派生数据系统**：派生系统中的数据，是对其他系统中已有数据进行转换或处理后的结果。如果派生数据丢失，可以从原始数据源重新构建。经典例子是缓存：命中时由缓存返回，未命中时回退到底层数据库。反规范化值、索引、物化视图、变换后的数据表示，以及在数据集上训练出的模型，都属于这一类。
- 在[“数据集成”](https://ddia.vonng.com/ch13/#sec_future_integration)中，我们会讨论如何组合多个数据系统，实现单一系统难以独立完成的能力。

#### 云服务的分层
eg：
- 使用 **对象存储** 服务存储大文件。提供了中间层，不用担心硬件问题。
- 在对象存储和其他云服务之上建立更多的服务，构建现有的高级别系统。
#### 存储与计算的分离
**传统方式**：存储和计算在同一台机器上，靠RAID保护数据。
**云环境的问题**：虚拟机的本地磁盘不持久，实例一换数据就丢。虚拟磁盘（如EBS）虽然能用，但慢且依赖网络，容易出问题。
**云原生做法**：
1. **存储与计算分离**——存储归存储（如S3），计算归计算，分开管理。
2. **用专用存储服务**——大文件放对象存储，小数据由数据库单独管理，不再依赖本地磁盘。（更清晰地说，对象存储不是单机磁盘，而是**分布式集群**）
3. **多租户共享**——多客户共用同一套硬件，节省成本，但必须做好隔离，防止互相影响。

### 分布式系统的问题
1. **网络不可靠** — 请求可能超时，不知道成功没有，重试有风险[第 9 章](https://ddia.vonng.com/ch9/#ch_distributed)
2. **性能差** — 网络调用比本地调用慢得多；节点多不一定更快，有时单机反而更快
3. **难排查** — 出问题不好找原因，需要追踪工具辅助
4. **一致性难** — 跨服务的数据同步很麻烦，分布式事务在微服务基本不用[第 8 章](https://ddia.vonng.com/ch8/#ch_transactions)
5. 如果你可以在单台机器上做某件事情，与搭建分布式系统相比通常要简单得多，成本也更低。[第 4 章](https://ddia.vonng.com/ch4/#ch_storage)
### 微服务与无服务器
**面向服务的体系结构**：分为客户端和服务器，并让客户端向服务器发出请求。最常见的是使用 HTTP 进行此通信。
**微服务**：将复杂应用拆分为多个小服务，每个服务有明确目的、独立团队维护、独立API。
**无服务器（Serverless）**： 把**代码执行也变成计量服务**，像用水用电一样，用多少付多少

## 描述性能
软件性能通常围绕两类指标展开：
- 响应时间：从用户发出请求到收到响应所经历的时间。单位是秒（或毫秒、微秒）。
- 吞吐量：系统每秒可处理的请求数或数据量。对于给定硬件资源，系统存在一个可处理的 _最大吞吐量_。单位是“每秒某种工作量”。
- 吐量和响应时间之间通常相关。在线服务的典型关系如 [图 2-3](https://ddia.vonng.com/ch2/#fig_throughput)：低吞吐量时响应时间较低，负载升高后响应时间上升。原因是 _排队_。请求到达高负载系统时，CPU 往往已在处理前一个请求，新请求只能等待；当吞吐量逼近硬件上限，排队延迟会急剧上升。

![](https://ddia.vonng.com/fig/ddia_0203.png)

图 2-3. 随着服务的吞吐量接近其容量，由于排队，响应时间急剧增加。
- 如果系统已接近过载、吞吐量逼近极限，有时会进入恶性循环：效率下降，进而更加过载。例如，请求队列很长时，响应时间可能高到让客户端超时并重发请求，导致请求速率进一步上升，问题持续恶化，形成 _重试风暴_。即使负载后来回落，系统也可能仍卡在过载状态，直到重启或重置。这种现象叫 _亚稳态故障_（Metastable Failure），可能引发严重生产故障
- 为了避免重试把服务拖垮，可以在客户端拉大并随机化重试间隔（_指数退避_），并临时停止向近期报错或超时的服务发请求（例如 _熔断器_ 或 _令牌桶_ ）。服务端也可在接近过载时主动拒绝请求（_负载卸除_ ），并通过响应要求客户端降速（_背压_）。此外，排队与负载均衡算法的选择也会影响结果。
- 若系统可通过增加计算资源显著提升最大吞吐量，就称其 _可伸缩_。（ [“可伸缩性”](https://ddia.vonng.com/ch2/#sec_introduction_scalability)）
### 延迟与响应时间
- _响应时间_ 是客户端看到的总时间，包含链路上各处产生的全部延迟。
- _服务时间_ 是服务主动处理该请求的时间。
- _排队延迟_ 可发生在流程中的多个位置。例如请求到达后，可能要等 CPU 空出来才能处理；同机其他任务若占满出站网卡，响应包也可能先在缓冲区等待发送。
- _延迟_ 是对“请求未被主动处理这段时间”的统称，也就是请求处于 _潜伏（latent）_ 状态的时间。尤其是 _网络延迟_（或网络时延）指请求与响应在网络中传播所花的时间。

图 2-4. 响应时间、服务时间、网络延迟和排队延迟。

## 可靠性与容错
### 可靠性
_可靠性_ 可以粗略理解为：即使出现问题，系统仍能持续正确工作。为了更精确地描述“出问题”，我们区分 _故障_ 与 _失效_：
- 故障：指系统某个 _局部组件_ 停止正常工作：例如单个硬盘损坏、单台机器宕机，或系统依赖的外部服务中断。
- 失效：指 _整个系统_ 无法继续向用户提供所需服务；换言之，系统未满足服务级别目标（SLO）。
- 比如一个硬盘坏了，对“硬盘这个系统”来说是失效；但对“由许多硬盘组成的更大系统”来说，它只是一个故障。更大系统若在其他硬盘上有副本，就可能容忍该故障。
### 容错
- 如果系统在发生某些故障时仍继续向用户提供所需的服务，我们称系统为 _容错的_。例如在社交网络案例中，扇出流程里可能有机器崩溃或不可用，导致物化时间线更新中断。若要让该流程具备容错性，就必须保证有其他机器可接管任务，同时既不漏投帖子，也不重复投递。（这个思想称为 _恰好一次语义_，我们会在 [“数据库的端到端论证”](https://ddia.vonng.com/ch13/#sec_future_end_to_end) 中详细讨论。）
- 如果系统不能容忍某个部分变得有故障，我们称该部分为 _单点故障_（SPOF），因为该部分的故障会升级导致整个系统的失效。
- 故意 _提高_ 故障发生率的方式叫做**故障注入**。通过主动注入故障，可以持续演练并验证容错机制，_混沌工程_ 就是围绕这类实验建立起来的方法论。
### 硬件与软件故障
在大规模系统里，硬件故障足够频繁，已经是“正常运行”的一部分。
#### 通过冗余容忍硬件故障

#### 软件故障
软件故障常常高度相关，因为许多节点运行同一套软件，也就共享同一批 bug。这类故障更难预判，也往往比“相互独立的硬件故障”造成更多系统失效 。

## 可伸缩性
_可伸缩性_ 用来描述系统应对负载增长的能力。
当你的产品真的做起来、负载持续上升时，你自然会看到瓶颈在哪里，也就知道该沿哪些维度扩展。那时再系统性投入可伸缩性技术，通常更合适。
### 描述负载
如果资源翻倍后能承载两倍负载且性能不变，这称为 _线性可伸缩性_，但更常见的是成本增长快于线性，低效原因也很多。比如数据量增大后，即使请求大小相同，处理一次写请求也可能比数据量小时更耗资源。
### 共享内存、共享磁盘与无共享架构
- 增加服务硬件资源的最简单方式，是迁移到更强的机器，购买（或租用）拥有更多 CPU 核心、更多 RAM、更多磁盘的实例。这叫 _纵向伸缩_（scaling up）。
- _共享内存架构_：同一进程内线程共享同一块 RAM，通过多进程/多线程获得并行性。
- _共享磁盘架构_：多台机器各有独立 CPU 和 RAM，但共享同一组磁盘阵列，通过高速网络连接（NAS 或 SAN）。该架构传统上用于本地数据仓库场景，但争用与锁开销限制了其可伸缩性。
- _无共享架构_（即 _横向伸缩_、scaling out）：使用多节点分布式系统，每个节点拥有自己的 CPU、RAM 和磁盘；节点间协作通过常规网络在软件层完成，一些云原生数据库把“存储”和“事务执行”拆成独立服务（参见 [“存储与计算分离”](https://ddia.vonng.com/ch1/#sec_introduction_storage_compute)），由多个计算节点共享同一存储服务。
- 无共享的优势在于：具备线性伸缩潜力、可灵活选用高性价比硬件（尤其在云上）、更容易随负载增减调整资源，并可通过跨多个数据中心/地域部署提升容错。代价是：需要显式分片（见 [第 7 章](https://ddia.vonng.com/ch7/)），并承担分布式系统的全部复杂性（见 [第 9 章](https://ddia.vonng.com/ch9/)）。
### 可伸缩性原则
- 能够大规模运行的系统架构，通常高度依赖具体应用，不存在通用“一招鲜”的可伸缩架构（俗称 _万金油_）。例如：面向“每秒 10 万次请求、每次 1 kB”的系统，与面向“每分钟 3 次请求、每次 2 GB”的系统，形态会完全不同，尽管二者数据吞吐量都约为 100 MB/s。
- 可伸缩性的一个通用原则，是把系统拆分成尽量可独立运行的小组件。这也是微服务（参见 [“微服务与无服务器”](https://ddia.vonng.com/ch1/#sec_introduction_microservices)）、分片（[第 7 章](https://ddia.vonng.com/ch7/)）、流处理（[第 12 章](https://ddia.vonng.com/ch12/#ch_stream)）和无共享架构的共同基础。难点在于：哪里该拆，哪里该合。另一个好原则是：不要把系统做得比必要更复杂。
## 可维护性
维护软件的适用原则：
- **可运维性（Operability）**：让组织能够更容易地保持系统平稳运行。
- **简单性（Simplicity）**：采用易理解且一致的模式与结构，避免不必要复杂性，让新工程师也能快速理解系统。
- **可演化性（Evolvability）**：让工程师在未来能更容易修改系统，使其随着需求变化而持续适配并扩展到未预料场景。
### 可运维性：让运维更轻松
良好的可运维性意味着把日常任务做简单，让运维团队把精力投入到高价值工作。数据系统可以通过多种方式达成这一点：
- 让监控工具能获取关键指标，并支持可观测性工具（参见 [“分布式系统的问题”](https://ddia.vonng.com/ch1/#sec_introduction_dist_sys_problems)）以洞察运行时行为。相关商业/开源工具都很多。
- 避免依赖单机（系统整体不停机的前提下允许下线机器维护）。
- 提供完善文档和易理解的操作模型（“我做 X，会发生 Y”）。
- 提供良好默认值，同时允许管理员在需要时覆盖默认行为。
- 适当支持自愈，同时在必要时保留管理员对系统状态的手动控制权。
- 行为可预测，尽量减少“惊喜”。
### 简单性：管理复杂度
- 管理复杂性，一种常见分析方法是把复杂性分成两类：**本质复杂性** 与 **偶然复杂性**。前者源于业务问题本身，后者源于工具与实现限制。但这种划分也并不完美，因为随着工具演进，“本质”和“偶然”的边界会移动。
- 管理复杂度最重要的工具之一是 **抽象**。例如，高级语言是对机器码、CPU 寄存器和系统调用的抽象。SQL 则抽象了磁盘/内存中的复杂数据结构、来自其他客户端的并发请求，以及崩溃后的不一致状态。
### 可演化性：让变化更容易
- **关键影响因素**：松耦合与简单性让系统更容易修改，反之则难。
- **重要实践**：尽量减少不可逆操作（如数据库迁移无法回切），降低变更风险，提升灵活性。
## 总结
**性能、可靠性、可伸缩性与可维护性**
# 3. 数据模型与查询语言
大多数应用程序都是通过层层叠加的数据模型来构建的，每一层通过提供一个简洁的数据模型来隐藏下层的复杂性。
- **核心地位**：数据模型影响深远，不仅决定代码怎么写，也塑造我们如何理解问题。
- **分层架构**：应用程序通过层层抽象构建，每一层都通过简洁的模型隐藏下层细节（如：现实世界 → 应用中的对象/API → 通用的JSON/关系表 → 存储引擎的字节 → 硬件）。
- **协作基础**：这种分层让不同分工（如数据库工程师与应用开发者）能有效协作。
- **不同模型有不同取舍**：关系模型、文档模型、图模型等各有擅长，没有“万能”模型。选择合适的模型，取决于具体的数据类型和查询需求。
## 关系模型与文档模型
- **关系模型（SQL）**：数据组织成表（行+列），结构严格，长期占据主导地位。
- **文档模型（JSON）**：数据以嵌套文档形式存储，模式灵活，近年兴起作为重要补充。
- 关系模型强调规范化和严格结构，文档模型追求灵活性和开发便捷性。两者不是简单替代，而是各有适用场景。
以下都是关于文档和关系数据的优缺点的辩论。
### 对象关系不匹配
如今，大部分应用程序开发都是使用面向对象的编程语言完成的（嵌套、多态、引用），但关系数据库数据以**表、行、列**形式组织（扁平、严格结构）。写代码时得在两者间来回转换——把对象拆成表存进去，查出来再拼回对象。这个转换层很繁琐，被称为**阻抗不匹配**。
#### 对象关系映射（ORM）
对象关系映射（ORM）框架（如 ActiveRecord 和 Hibernate）可以减少这个转换层所需的样板代码量，能把对象语言翻译成表语言。
- 缺点：容易产生低效查询、跨系统支持有限、无法完全隐藏差异...
- 优点：减少重复的转换代码、帮助做缓存、辅助管理数据库变更（迁移）

### 规范化、反规范化与连接
- 对于"region_id": "us:91"而言，数据更加 _规范化_；对人类有意义的信息（如文本 _Washington, DC_）而言，这种表示是 _反规范化_ 的。规范化可以保持ID不变，但是需要多一层查找在关系数据模型中，这是使用 _连接_ 完成的，例如：

```sql
SELECT users.*, regions.region_name
    FROM users
    JOIN regions ON users.region_id = regions.id
    WHERE users.id = 251;
```
- 因为一些文档数据库根本不支持连接，并且JSON 数据模型使得存储额外的反规范化字段变得容易，它们通常与反规范化相关联。你必须在应用程序代码中执行连接。
#### 规范化的权衡
- 规范化数据通常写入更快（因为只有一个副本），但查询更慢（因为它需要连接）；反规范化数据通常读取更快（连接更少），但写入更昂贵（更多副本要更新，使用更多磁盘空间）。有时候需要将反规范化视为派生数据的一种形式。（[“记录系统与派生数据”](https://ddia.vonng.com/ch1/#sec_introduction_derived)）
- 并非所有数据库都在多个文档之间提供原子性。可以通过流处理确保一致性（ [“保持系统同步”](https://ddia.vonng.com/ch12/#sec_stream_sync) ）。
- 小数据更适合规范化（OLTP系统），大数据更适合反规范，因为连接的成本可能会成为问题。
#### 社交网络案例研究中的反规范化
反规范化的缺陷：
- **数据可能不一致**：同一份数据存了多份，改的时候容易漏改，导致新旧数据同时存在。
-  **写入更慢、更贵**：发一条帖子，可能要去更新几百万个粉丝的缓存，俗称“写放大”。
-  **浪费存储空间**：存了大量重复数据。
这时使用规范化的ID更加合理

### 星型与雪花型：分析模式
[图 3-5](https://ddia.vonng.com/ch3/#fig_dwh_schema) 显示了一个可能在杂货零售商的数据仓库中找到的**星型模式**示例。模式的中心是所谓的 _事实表_（在此示例中，它称为 `fact_sales`）。事实表的每一行代表在特定时间发生的事件（这里，每一行代表客户购买产品）。如果我们分析的是网站流量而不是零售销售，每一行可能代表用户的页面查看或点击。通常，事实被捕获为单个事件，因为这允许以后最大的分析灵活性。然而，这意味着事实表可能变得非常大。

![](https://ddia.vonng.com/fig/ddia_0305.png)

图 3-5. 用于数据仓库的星型模式示例。
- 事实表中的一些列是属性，例如产品售出的价格和从供应商那里购买它的成本（允许计算利润率）。**事实表中的其他列是对其他表的外键引用**，称为 _维度表_。由于事实表中的每一行代表一个事件，维度代表事件的 _谁_、_什么_、_哪里_、_何时_、_如何_ 和 _为什么_。例如，在 [图 3-5](https://ddia.vonng.com/ch3/#fig_dwh_schema) 中，其中一个维度是售出的产品。`dim_product` 表中的每一行代表一种待售产品类型，包括其库存单位（SKU）、描述、品牌名称、类别、脂肪含量、包装尺寸等。`fact_sales` 表中的每一行使用外键来指示在该特定交易中售出了哪种产品。查询通常涉及对多个维度表的多个连接。
- 当表关系被可视化时，事实表位于中间，被其 _维度表_ 包围；到这些表的连接就像星星的光芒。
- 星型模式一个变体被称为 _雪花模式_，其中维度被进一步分解为子维度。例如，品牌和产品类别可能有单独的表，`dim_product` 表中的每一行都可以将品牌和类别作为外键引用，而不是将它们作为字符串存储在 `dim_product` 表中。
- 星型或雪花模式主要由多对一关系组成（例如，许多销售发生在一个特定产品，在一个特定商店），表示为事实表对维度表的外键，或维度对子维度的外键。
- 一些数据仓库模式进一步进行反规范化，完全省略维度表，将维度中的信息折叠到事实表上的反规范化列中（本质上是预计算事实表和维度表之间的连接）。这种方法被称为 _一张大表_（OBT），虽然它需要更多的存储空间，但有时可以实现更快的查询。
### 何时使用哪种模型
**文档模型：**
- 数据天生就是**一对多**的树状结构（比如一个人的多份工作、多个学历）。
- 需要**把数据放一起**一次性读取（局部性好，性能高）。
- 字段经常变化，需要**模式灵活**。
#### 文档模型中的模式灵活性
- **读时模式（文档模型）**：写入时不检查格式，读取时才解释。类似编程语言的**动态类型**（运行时才确定）。
- **写时模式（关系模型）**：写入时必须符合预定义格式。类似**静态类型**（编译时检查）。
- 灵活性：
- **改格式不用重写所有数据**
    - 比如把“全名”字段拆成“名”和“姓”。
    - 文档模型：老数据留着，代码里判断一下就行（`if (!first_name) {从name解析}`）。
    - 关系模型：需要执行 `ALTER TABLE` + `UPDATE`，大表上可能非常慢。
- **适合异构数据**
    - 不同类型的对象混在一起（比如日志、配置、外部系统的数据）。
    - 不需要为每种类型单独建表。
#### 读写的数据局部性
- 文档通常存为一个连续字符串（如JSON），这种 _存储局部性_ 具有性能优势。如果你经常需要一次性读取**整个文档**（比如渲染一个网页），这种方式很快，因为只需要一次磁盘寻道。
- 但是数据库通常得加载**整个文档**。如果你只访问其中一小部分，就会浪费内存和带宽，修改文档的一小部分，也常常要**重写整个文档**。因此文档不宜太大，也不宜频繁做小改动。
#### 文档的查询语言
文档数据库更加多样化，例子：象你是一名海洋生物学家，每次你在海洋中看到动物时，你都会向数据库添加一条观察记录。现在你想生成一份报告，说明你每个月看到了多少条鲨鱼。在 PostgreSQL 中，你可能会这样表达该查询：

```sql
SELECT date_trunc('month', observation_timestamp) AS observation_month, ❶ 
    sum(num_animals) AS total_animals
FROM observations
WHERE family = 'Sharks'
GROUP BY observation_month;
```

❶ : `date_trunc('month', timestamp)` 函数确定包含 `timestamp` 的日历月，并返回表示该月开始的另一个时间戳。换句话说，它将时间戳向下舍入到最近的月份。
此查询首先过滤观察结果以仅显示 `Sharks` 家族中的物种，然后按它们发生的日历月对观察结果进行分组，最后将该月所有观察中看到的动物数量相加。可以使用 MongoDB 的聚合管道表达相同的查询，如下所示：

```
db.observations.aggregate([
    { $match: { family: "Sharks" } },
    { $group: {
    _id: {
        year: { $year: "$observationTimestamp" },
        month: { $month: "$observationTimestamp" }
    },
    totalAnimals: { $sum: "$numAnimals" }
    } }
]);
```

聚合管道语言在表达能力上类似于 SQL 的子集，但它使用基于 JSON 的语法而不是 SQL 的英语句子风格语法；
