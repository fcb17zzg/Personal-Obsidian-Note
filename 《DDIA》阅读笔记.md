# 前言
**[[数据密集型]]**： 数据是其主要挑战（数据量，数据复杂度或数据变化速度）。
**[[计算密集型]]**：处理器速度是其瓶颈。
本书的目的：尝试寻找 **思考** 数据系统的有效方式 —— 不仅关于它们 **如何** 工作，还包括它们 **为什么** 以这种方式工作，以及哪些问题是我们需要问的。阅读本书后，你能很好地决定哪种技术适合哪种用途，并了解如何将工具组合起来，为一个良好应用架构奠定基础。
### 本书的目标读者
- 你想了解如何使数据系统可伸缩，例如，支持拥有数百万用户的 Web 或移动应用。
- 你需要提高应用程序的可用性（最大限度地减少停机时间），保持稳定运行。
- 你正在寻找使系统在长期运行过程易于维护的方法，即使系统规模增长，需求与技术也发生变化。
- 你对事物的运作方式有着天然的好奇心，并且希望知道一些主流网站和在线服务背后发生的事情。这本书打破了各种数据库和数据处理系统的内幕，探索这些系统设计中的智慧是非常有趣的。
### 本书涉及的领域
- 数据系统的 **架构（architecture）**，以及它们被集成到数据密集型应用中的方式。
- **大数据（Big Data）**，如 “单节点” 之于 “分布式系统”，或 “在线 / 交互式系统” 之于 “离线 / 批处理系统”。
# 第一部分：数据系统基础
## 数据系统架构中的权衡
- [[数据密集型]]：
	- 存储数据，以便它们或其他应用程序以后能再次找到（**数据库**）
	- 记住昂贵操作的结果，以加快读取速度（**缓存**）
	- 允许用户按关键字搜索数据或以各种方式过滤数据（**搜索索引**）
	- 一旦事件和数据变更发生就立即处理（**流处理**）
	- 定期处理累积的大量数据（**批处理**）
- [[后端]]服务通常通过 [[HTTP]]（有时是 WebSocket）提供访问。其核心是应用代码：在一个或多个数据库中读写数据，并按需接入缓存、消息队列等其他系统（可统称为 **数据基础设施**）。应用代码往往是 **无状态** 的：处理完一个 HTTP 请求后，不保留该请求上下文。因此，凡是需要跨请求持久化的信息，都必须写在客户端，或写入服务器端数据基础设施。
### 分析型与事务型系统
- **[[事务型系统]]** 由后端服务和数据基础设施组成，在这里创建数据，例如通过服务外部用户。在这里，应用程序代码基于用户执行的操作读取和修改其数据库中的数据。
- **[[分析型系统]]** 服务于业务分析师和数据科学家的需求。它们包含来自事务型系统的只读数据副本，并针对分析所需的数据处理类型进行了优化。
### 事务处理与分析的特征
- **[[联机事务处理]]**：基于用户的输入插入、更新或删除记录。
- **[[联机分析处理]]**：扫描大量记录，并计算聚合统计信息（如计数、求和或平均值），而不是将单个记录返回给用户。
- 表 1-1. 事务型系统和分析型系统特征比较

|属性|事务型系统（OLTP）|分析型系统（OLAP）|
|---|---|---|
|主要读取模式|点查询（通过键获取单个记录）|对大量记录进行聚合|
|主要写入模式|创建、更新和删除单个记录|批量导入（ETL）或事件流|
|人类用户示例|Web 或移动应用程序的最终用户|内部分析师，用于决策支持|
|机器使用示例|检查操作是否被授权|检测欺诈/滥用模式|
|查询类型|固定的查询集，由应用程序预定义|分析师可以进行任意查询|
|数据代表|数据的最新状态（当前时间点）|随时间发生的事件历史|
|数据集大小|GB 到 TB|TB 到 PB|
### 数据仓库
**一个专门用于分析的独立数据库**，它通过 **ETL（提取-转换-加载）** 过程，从企业分散的各类事务处理系统（OLTP）中获取数据的只读副本，并进行清洗和转换，使分析师可以在不影响公司日常业务操作的情况下，方便地查询和分析来自不同源系统的数据，以支持决策。
![](https://ddia.vonng.com/fig/ddia_0101.png)

图 1-1. ETL 到数据仓库的简化概述。
**OLTP 采用的“实体-关系模型”（3NF，即第三范式）是为了高效处理事务**，它通过将数据拆解成大量离散的实体表（如用户、订单、产品、地址），并用外键关联，来确保数据无冗余、更新一致、插入删除快。但这恰恰是分析的“死穴”——分析查询往往需要**聚合、分组、排序**，意味着要同时关联（JOIN）很多张表，扫描海量数据行。表越多、数据量越大，这类查询就越慢、越复杂，甚至无法在合理时间内完成。
而数据仓库采用的“[[星型模型]]”和“[[雪花型模型]]”就是为了分析而设计的：
- **星型模型**：将数据组织成一个 **“事实表”**（记录业务事件，如每一笔销售）和多个 **“维度表”**（记录业务对象，如时间、产品、门店）。事实表通过外键链接维度表。这种模式是非规范化的，维度表里会包含很多冗余的描述信息（如产品名称、类别、品牌都放在一张产品表里）。
    - **为什么适合分析？** 因为一个典型的分析查询（如“按产品类别统计去年每个季度的总销售额”）只需要关联**一张事实表**和**一到两张维度表**。JOIN 数量少，查询更简单、更快。
- **雪花型模型**：是星型模型的变种，它会进一步将维度表**规范化**，拆分成更细的子维度表。例如，将产品维度表拆分成产品、品牌、类别三张独立的表。
    - **为什么存在？** 雪花型比星型**节省存储空间**，并且对某些维度层次变化的维护更友好。
    - **为什么不是主流？** 对于分析查询来说，雪花型引入了**更多的关联操作**，查询性能通常不如星型模型。因此，在追求查询速度和简洁性的现代数据仓库（如 Google BigQuery, Amazon Redshift）中，星型模型（甚至更宽、更非规范化的表）更为常见。
#### 从数据仓库到数据湖
- **数据湖**：一个集中式存储库，用于存放任何类型（不限于关系型数据）的原始数据副本。
	- **无强制模式**：与数据仓库不同，数据湖不强制特定的文件格式或数据模型，可以存储文本、图像、视频等任意格式的数据。
	- **成本较低**：通常基于廉价的对象存储（如云存储）构建。
	- **面向数据科学**：主要为了满足数据科学家对**特征工程**、**自然语言处理**等非SQL任务的需求。
	- **遵循“寿司原则”**：保存最原始的数据，让每个数据消费者根据自己的需要进行转换。
- **数据湖仓**：结合了数据湖的灵活性和数据仓库的分析能力，允许直接在数据湖文件上运行SQL查询。
- **传统 ETL** x **数据管道**
	-  **传统 ETL**：是数据管道的一种**特定、早期的实现形式**，主要服务于数据仓库。
	- **数据管道**：是一个**更通用、更现代的概念**，覆盖了从简单数据复制到复杂实时流处理的各种数据移动和处理流程，是构建数据湖和数据湖仓的核心组件。

### 记录系统与派生数据
- **权威记录系统**：记录系统，也称 **真相来源（权威数据源）**，保存某类数据的权威（canonical）版本。新数据进入系统时（例如用户输入）首先写入这里。每个事实只表示一次（这种表示通常是 **规范化** 的；见[“规范化、反规范化与连接”](https://ddia.vonng.com/ch3/#sec_datamodels_normalization)）。如果其他系统与记录系统不一致，则按定义以记录系统为准。
- **派生数据系统**：派生系统中的数据，是对其他系统中已有数据进行转换或处理后的结果。如果派生数据丢失，可以从原始数据源重新构建。经典例子是缓存：命中时由缓存返回，未命中时回退到底层数据库。反规范化值、索引、物化视图、变换后的数据表示，以及在数据集上训练出的模型，都属于这一类。
- 在[“数据集成”](https://ddia.vonng.com/ch13/#sec_future_integration)中，我们会讨论如何组合多个数据系统，实现单一系统难以独立完成的能力。
