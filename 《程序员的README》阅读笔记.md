## 玩转代码
- 软件的熵：代码混乱走向无须的趋势，[[技术债]]是造成的主要原因之一
- [[技术债]]矩阵：

|     | 鲁莽的       | 谨慎的           |
| --- | --------- | ------------- |
| 有意的 | ”没时间设计“   | ”先发布再处理后续“    |
| 无意的 | ”什么是分层结构“ | ”现在知道了当时改怎么做“ |
- 解决[[技术债]]：如何讨论？1.按事实陈述 2.描述风险和成本 3.提出方案 4.备选方案 5.权衡利弊
- 如何[[变更代码]]：1.定义变更点（埋种子的地方） 2.找到[[测试]]点（打破依赖关系->将大的方法拆分成多个小方法；引入一个接口或中介；注入明确的控制点，如时间的推移）3.过手的代码要比之前的更干净（包括去除代码异味），持续重构工程，将清理代码的提交和改变行为的提交分开 4.做渐变式修改 5.务实重构
- 如何“避坑”：1.保守一些的技术选型，避免缺乏成熟度的新技术，应该把“代币”花在高价值领域（核心竞争力）的技术上 2.不要特立独行，顺应公司标准 3.不要只分叉而不向上游提交修改  4.克制重构的冲动，小心过度扩张
## 编写可维护的代码
- 防御式编程：
	- 避免空值，在开头做空值检查，使用空对象模式
	- 保持变量不可变（const）
	- 使用类型提示和静态类型检查器
	- 验证输入
	- 善用异常，异常要有精确含义
	- 早抛晚捕（在尽可能接近错误的地方引发异常，在调用的堆栈上传播这个异常，直到能够处理异常的程序的层级）
	- 智能重试：在系统或网络请求失败时，不简单地立即重试或固定间隔重试，而是根据失败原因、系统状态、业务特性等因素，**动态、智能地决定是否重试、何时重试、重试多少次**的一套机制
	- 构建幂等系统，及时释放资源
- [[日志]]使用：
	- 给日志分级:TRACE:逐行日志/DEBUG:在产品出故障时有用，正常操作没用/INFO:正常运转/WARN/ERROR/FATAL:遇到非常严重的情况，类似core文件
	- 原子日志：在一行消息中包含所有相关消息，不要折行
	- 关注日志性能（字符串拼接、异步附加器，改变日志的冗余度和配置可以消除竞争条件的bug）
- 系统监控
	- 用各种系统指标监控你的应用程序：计数器、仪表盘和直方图
	- 使用标准的监控组件
	- 测量一切：**资源池**、[[缓存]]、数据结构、CPU I/O密集型操作、数据大小、异常和错误、远程请求和响应
- 跟踪器：分布式调用跟踪（对上游API的调用可能会导致下游的数百次不同服务的RPC调用）->附加一个调用ID
- 配置相关注意事项、工具集

## 依赖管理
- 依赖管理的基础知识
	- 好的版本管理方案，其版本的特点：唯一性（版本不应该重复使用，不要再现有版本下重新发布更改的代码）、可比性（帮助人们和工具对版本的优先顺序进行推断）、信息性（语义版本管理：主版本号+次版本号+补丁版本号，次版本号向下兼容，还可以添加一个“-”来定义预发布版本，附加一个流水号如2.13.7-alpha.2+1942）
	- 获取依赖树
- 相依性地狱：钻石依赖（在同一个类库使用两个不同版本）、循环依赖、版本冲突
- 如何避免：隔离依赖项、按需添加依赖项、指定依赖项的版本（界定版本范围）、依赖范围最小化、保护自己免受循环依赖的影响
## [[测试]]
- 测试类型：
	- 单元测试：验证某个单一的方法或行为
	- 集成测试：验证多个组件在一起后是否还能工作
	- 系统测试：验证整个系统的整体运行情况
	- 性能测试
	- 验收测试
- 测试工具：
	- 测试用例的编写工具（模拟库）：可以消除外部依赖性，但使用时要考虑是否可以重构代码以消除对模拟系统的依赖
	- 测试框架
	- 代码质量工具
- 动手编写测试
	- 编写干净的测试代码，不要与实现的细节贴合太紧，这样对代码主体的改变会破坏测试代码，将测试的依赖项和常规代码的依赖项分开
	- 避免过度测试，高的代码覆盖率不能保证正确性，首先关注高风险的区域
## 代码评审
- 被评审：
	- 准备：描述性的提交信息，注释和测试，未解决的问题或实现细节标注
	- 用评审草案降低风险
	- 提交评审请勿触发测试
	- 预排大体量的代码修改
	- 不要太在意批评性的评论
	- 保持同理心，但不要容忍粗鲁
	- 保持主动
- 评审：
	- 分流评审请求（关键/不关键）
	- 给评审预留时间
	- 理解修改的意图
	- 提供全面的反馈
	- 承认优点，区分问题、建议和挑刺
	- 不要只做橡皮图章
	- 不要局限于使用网页版的评审工具
	- 不要忘记评审测试代码
	- 推动决断
## 软件交付
软件交付流程：**构建、发布、部署和展开**
#### 分支策略
- 分支应该可以快速合并到主分支，频繁的合并称为**持续集成**
- 当分支不稳定以至于无法发布给用户时，或者开发人员希望避免进入特性冻结期，即在主分支线稳定后禁止提交特性时，基于**特性分支**的开发就很常见。
- Gitflow：使用开发分支、**热修复分支**和发布分支
#### 构建环节
- 打包需要带版本号
- 将不同的资源（代码、配置、schema、图像和语言包等）单独打包
#### 发布环节
- 不要程序只交付一半
- 将包发布到仓库（VCS、Docker Hub、PyPI...）
- 保持版本不变性
- 频繁发布
- 对发布计划保持透明（每季度或一年发布一次）
- 撰写变更日志和发行说明
#### 部署环节
- 自动部署，使用脚本而不是人工
- 部署的原子性，在与旧版本不同的位置上安装软件，不要覆盖任何东西
- 独立地部署应用，务必避免顺序部署的要求
#### 展开环节
- 系统监控：对系统各项指标（如CPU、内存、磁盘、网络、应用响应时间、错误率等）进行实时采集、展示和告警，用于发现问题、定位根因和评估系统健康状态。
- 特性开关：通过配置动态开启或关闭某个功能，无需重新发布代码；支持灰度发布、A/B测试和快速回滚有问题的特性。
- 熔断器：当依赖服务错误率或超时率达到阈值时，自动切断请求，避免级联故障；经过冷却时间后，允许少量请求探测服务是否恢复。
- 并行的服务版本梯队：同时运行多个版本的服务实例（如V1、V2），流量按规则分配；便于灰度上线、平滑迁移和快速回退。
- 抹黑启动：新版本启动时，先将其置于“黑暗”模式——接收并处理真实流量，但不返回给调用方或影响业务结果；用于验证新版本的正确性和性能，而不影响线上用户。
## On-Call
#### On-Call技能包
- 随时响应
- 保持专注
- 确定工作优先级（P1、P2...）
- 清晰的沟通（回应请求迅速）
- 跟踪你的工作，总结成文档
#### 事故处理
* 分流：确定优先级
* 协同：将事故通知所有相关方，交流问题，给出下一次更新反馈的预估时间，并在调查过程中向请求者提供最新信息
* 应急方案：降低问题的影响，通常是将软件回滚或将流量从问题上转移开
* 解决方案：花时间来排除故障并解决根本问题（在更大的系统上使用分片检索或二分法搜索，即在堆栈的一般位置上设置一个断点，看看问题在上游还是在下游）
* 后续行动：起草总结文档，安排与所有相关方的评审会议
* 不要逞英雄
## 技术设计流程
软件设计是在独立工作和相互合作之间交替进行的螺旋式上升的过程，每一步都要明确和完善。
#### 关于设计的思考
- 定义问题：把相关利益人的反馈整理成清晰的问题陈述。“如果我们不解决这个问题会怎么样”，有许多问题其实并不需要解决。要注意问题的范围。
- 着手调查：考虑相关的研究，替代的解决方案，以及权衡各方案的利弊，看看别人是怎么解决这个问题的，并进行批判性思考。
- 进行试验：编写API草案和部分实现，运行性能测试，甚至A/B用户测试以了解系统和用户的行为。
- 给些时间：设计需要深度思考。
#### 撰写设计文档
- 文档持续变更，需要设计文档的标准：
	- 该项目需要至少一个月的工程时间
	- 这一变更将对软件的扩展和维护产生长期的影响
	- 该变更将显著影响其他团队
- 了解撰写文档的目的：帮助你思考、获得反馈、让你的团队了解情况、培养新的工程师，并推动项目规划。
- 学会写作：文档要简明扼要，要让目标受众读懂你写的内容。
- 保证文档是最新的
#### 使用设计文档模板
摘要、现状与背景、变更的目的、需求（面向用户、技术、安全性和合规性...）、潜在的解决方案、建议的解决方案、设计与架构（系统构成图、UI/UX变更点、代码变更点、API变更点、持久层变更点）、测试计划、发布计划、遗留的问题、附录
#### 协作设计
- 理解你的团队的设计评审流程：架构评审委员会（外部人员，大型变更）和“请求裁定”（团队内部，快速）
- 不要让人惊讶
- 用设计讨论来进行头脑风暴
- 为设计出力
## 构建可演讲的架构
复杂系统的特点：**高依赖性**和**高隐蔽性**、**高惯性**
#### 可演进的设计
KISS原则->保持简单、愚蠢，使用最小惊讶原则和封装原则
- 你不是真的需要，避免过早优化，避免不必要的灵活抽象模型，以及避免最小可行产品所不需要的产品特性。
- **蒙茨法**：对于所构建的一切，问问哪些是绝对必要的，其余的就舍弃掉
- 可以在你怀疑可以插入优化的地方放置接口填充程序，但不要真正实现它们。
- **最小惊讶原则**：不要让用户感到惊讶，构建特性表现得要像用户最初期望的那样，也不要构建复杂代码让开发者惊讶。切记让参数需求具体化和可视化。
- 封装专业领域知识：高内聚、低耦合
#### 可演进的API
- 保持API小巧，应用**YANGI**哲学：只添加即刻需要的API方法或字段
- 公开定义良好的服务端API，使用现有的API定义框架
- 保持API变更的兼容性（向前兼容和向后兼容）
- API版本化（和文档一起）
#### 可持续的数据管理
- 数据库隔离
- 使用schema，不要隐藏无模式的数据，可以暂时让数据无模式化用以缓解向新的显示schema过渡。
- schema的自动迁移
- 保持schema的兼容性
## 敏捷计划
第一原则：**个人和互动**高于流程和工具
#### 敏捷计划的框架
- **Scrum**：鼓励短期迭代，并经常设有检查点来调整计划，开发工作被划分成几个阶段。
- **看板**：为每个工作流程阶段设置了垂直列的仪表盘，由标题框代表的任务，随着状态的变化在各列之间移动。
#### Scrum框架
- 用户故事：作为一名<用户>，我<想><这样>（工作票）
- 任务分解：作为一名开发人员，我想将用户故事拆解为可独立交付的技术任务，以便精准估时和并行开发。
- 故事点：作为一名团队成员，我想用故事点为任务分配相对工作量，以便跨迭代对比团队速率。
- 消化积压：作为产品负责人，我会在冲刺前与团队一起评审并排列待办事项的优先级，以便确保每次冲刺只聚焦最有价值的需求。
- 冲刺计划：作为团队，我们共同选定本次冲刺要完成的积压项并拆解出具体行动，以便形成可执行的承诺。
- 站会：作为一名执行者，我每天用15分钟同步“昨天完成、今天计划、遇到障碍”，以便快速协调并消除阻塞。
- 评审机制：作为利益相关方，我在冲刺结束时验收可演示的成果，以便获取真实反馈并决定下一步调整。
- 回顾会：作为团队内部成员，我反思“做得好、做得不好、改进项”，以便在下个冲刺中持续优化流程。
- 路线图：作为决策者，我需要一个跨越3-6个月的功能演进视图，以便对齐战略目标并管理外部依赖。
## 与管理者合作
管理者是普通工程师和负责商业决策的高管之间的沟通渠道，向上管理对于获得资源（资金和工程师）以及确保你的团队得到认可、赞赏和倾听至关重要。
#### 沟通、目标与成长
- 一对一面谈：制定议程，并在一对一面谈中承担大部分谈话，实现把问题发给管理者，维护面谈文档，用下面的提示作为讨论的主题：
	- 大局观：公司方向、组织变革
	- 反馈：我需要改进的地方、对团队计划流程有什么看法、最大的技术难题、希望我能做什么而我做不到、最大的问题、公司最大的问题、我和团队遇到了什么阻碍...
	- 职业生涯：管理者对我有哪些建议、我有哪些可以改进的地方、希望自己有哪些技能、长期目标
	- 个人事务：建立联系
	- 如果没有收到面谈，问问管理者
- **PPP**（进展、计划与问题）
- **OKR**（每个季度1到3个）
- [[绩效]]考核
	- 自我评价：务必保有一份最新的在整个一年中完成的工作的清单来唤起记忆
	- 如果感到惊讶请与管理者讨论沟通上的偏差
	- 管理者应该在一年中经常给予反馈，如果没有应该主动询问
#### 向上管理
- 接受反馈：试着把管理者的反馈纳入你的观点，而不是直接采用，并且给予反馈的反馈
- 给予反馈：团队、公司、行为、项目、技术计划...使用情况、行为，影响框架
- 讨论你的目标
- 事情不顺时要采取行动（与管理者、HR部门沟通）
## 职业生涯规划
- T型人才：在大多数领域内都能有效地工作（前端、后端、运维、数据仓库、机器学习...）
